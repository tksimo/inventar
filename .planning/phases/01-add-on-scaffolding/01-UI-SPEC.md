# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Add-on Scaffolding.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

---

## Scope Note

Phase 1 delivers infrastructure only. The UI deliverable is a production-quality React SPA navigation shell with three stub routes (Inventory, Shopping List, Settings). Each route renders the section title in an empty content area. No inventory data, no forms, no feature interactions exist yet.

The design contract defined here establishes the token set and patterns that Phase 2 builds on directly. Every value here is a commitment — Phase 2 must not redefine these foundations.

| Label | 12px | 600 | 1.4 | Nav item labels (active and inactive), secondary metadata |

**Declared sizes:** 12, 14, 20 (3 sizes)

Note: Display size (28px) is out of scope for Phase 1. It will be declared in Phase 2 if a display-level heading is needed.

## Color

Phase 1 serves inside the Home Assistant ingress iframe. HA uses a dark theme by default. The app must not visually clash with the HA shell surrounding it.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #1C1C1E | App background, page surface — matches HA's dark shell |
| Secondary (30%) | #2C2C2E | Nav sidebar background, stub content area background |
| Accent (10%) | #3B82F6 | Active nav item indicator only |
| Destructive | #EF4444 | Not used in Phase 1 — reserved for Phase 2 delete actions |
| Text primary | #F2F2F7 | All primary text on dark backgrounds |
| Text secondary | #8E8E93 | Nav labels (inactive), secondary metadata |
| Border | #3A3A3C | Nav sidebar border, content area divider |

Accent reserved for: **active nav item left border indicator only**. The accent is a 3px left border on the currently active nav item (`aria-current="page"`). It is not applied to buttons, links, icons, or any other element in Phase 1.

**Color justification:** HA's default dark theme uses approximately `#1C1C1E` (iOS system gray 6) for panel backgrounds and `#2C2C2E` for sidebar. Matching this eliminates the visual seam between the HA shell and the embedded app. This palette can be overridden in Phase 2 when shadcn tokens are introduced.

## Component Inventory

These are the only UI elements Phase 1 must implement. No additional components.

### AppLayout

**Structure:** Fixed left sidebar (220px) + scrollable main content area (fills remaining width)

**Sidebar contains:**
- App name "Inventar" as text heading (20px, weight 600, text-primary)
- Three nav items (Inventory, Shopping List, Settings) stacked vertically
- Each nav item: icon (20px) + label (12px, weight 600) + 44px min-height touch target

**Main content area contains:**
- Page heading (20px, weight 600) matching the active route name
- Empty body — no content in Phase 1

**Responsive behavior:** Phase 1 is desktop-first (HA sidebar panel). Mobile layout is deferred to Phase 2. On viewports below 600px, the sidebar collapses to icon-only: icons remain visible; nav labels are hidden via CSS (`sr-only` class or `visibility: hidden`) so they remain in the DOM and accessible to screen readers on mobile HA access. Do NOT remove nav label elements from the DOM on narrow viewports.

### NavItem

**States:**
- Default: `text-secondary` label, `text-primary` icon
- Hover: `secondary` background, `text-primary` label
- Active (`aria-current="page"`): `accent` left border (3px), `text-primary` label and icon, `secondary` background

**Transition:** `background-color 150ms ease` on hover. No other transitions.

### StubPage

**Structure:** Page heading + placeholder paragraph

**Heading:** Route name (e.g. "Inventory"), 20px, weight 600, `text-primary`
**Placeholder paragraph:** Phase-specific note (see Copywriting Contract), 14px, weight 400, `text-secondary`

## Interaction Contract

| Interaction | Behavior |
|-------------|----------|
| Nav item click | React Router `<Link>` — client-side navigation, no page reload |
| Active nav detection | `aria-current="page"` via react-router-dom `NavLink` `end` prop on Inventory |
| Browser back/forward | Handled by react-router-dom browser history |
| Route not found | Redirect to `/` (Inventory stub) — no 404 page in Phase 1 |
| Initial load | Default route is `/` → Inventory stub |
| Keyboard navigation | Nav items are `<a>` elements (via NavLink) — browser default tab order applies |
| Focus visible | Browser default focus ring — do not suppress `outline` in CSS |

**No loading states in Phase 1.** There is no async data fetching. The `/healthz` endpoint is backend-only and not surfaced in the SPA.

## Copywriting Contract

| Element | Copy |
|---------|------|
| App name (sidebar) | Inventar |
| Nav item 1 | Inventory |
| Nav item 2 | Shopping List |
| Nav item 3 | Settings |
| Inventory stub body | Your inventory items will appear here. |
| Shopping List stub body | Your shopping list will appear here. |
| Settings stub body | App settings will appear here. |
| Primary CTA | Not applicable — Phase 1 has no CTAs |
| Empty state heading | Not applicable — Phase 1 stubs are not true empty states |
| Error state | Not applicable — Phase 1 has no data fetching |
| Destructive confirmation | Not applicable — Phase 1 has no destructive actions |

**Stub copy rationale:** The body text on each stub uses future tense ("will appear here") rather than a loading message or error message. This correctly communicates that the feature is not yet available, not that something failed. It also avoids placeholder text (lorem ipsum) that could confuse a user who installs the add-on early.

## HTML/Accessibility Contract

- `<nav>` element wraps the sidebar navigation
- `<ul>` + `<li>` list for nav items
- `<main>` element wraps the content area
- `<h1>` for the page heading in each stub route
- `aria-current="page"` on the active NavLink (react-router-dom NavLink handles this automatically)
- `aria-label="Main navigation"` on the `<nav>` element
- All nav icons must have `aria-hidden="true"` (decorative — label provides the accessible name)
- Nav labels hidden on narrow viewports via CSS only (`sr-only` / `visibility: hidden`) — label elements remain in the DOM so screen readers on mobile can still announce nav item names
- No `role` overrides unless required by a specific deviation from semantic HTML

---

## Asset Contract

- No custom fonts — system font stack only
- No images — Phase 1 has no product images or illustrations
- No external CDN resources — all assets are bundled by Vite
- Lucide icons imported individually (tree-shaken): `House`, `ShoppingCart`, `Settings` from `lucide-react`
- Icon size: 20px (`size={20}` prop) uniformly across all nav items

**Favicon:** Use `mdi:package-variant-closed` SVG inline as favicon (matches `panel_icon` in config.yaml). Render as a simple 32x32 SVG with `#3B82F6` fill.

## Vite / Build Contract

These constraints flow from CONTEXT.md D-06 and are repeated here as UI implementation requirements:

- `base: "./"` in `vite.config.js` — mandatory, not optional. All asset paths must be relative.
- Output directory: `frontend/dist/` — FastAPI mounts this via `StaticFiles`
- No absolute paths in any import or asset reference
- CSS must be bundled (not externally referenced) — Vite handles this by default
