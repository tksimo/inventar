    Step B — Create `backend/routers/ha_display.py` exactly per 06-RESEARCH.md "Complete router skeleton":
    ```python
    """Public REST summary endpoint for HA REST-sensor polling (HA-01, D-01, D-03)."""
    from __future__ import annotations
    from fastapi import APIRouter, Depends
    from sqlalchemy.orm import Session
    from db.database import get_db
    from models import Item, QuantityMode, StockStatus
    from schemas.ha_display import HASummaryResponse

    router = APIRouter(prefix="/api/ha", tags=["ha-display"])

    @router.get("/summary", response_model=HASummaryResponse)
    def ha_summary(db: Session = Depends(get_db)) -> HASummaryResponse:
        active = db.query(Item).filter(Item.archived == False).all()  # noqa: E712

        out_of_stock = [
            i for i in active
            if (i.quantity_mode == QuantityMode.STATUS and i.status == StockStatus.OUT)
            or (i.quantity_mode == QuantityMode.EXACT and i.quantity is not None and i.quantity == 0)
        ]
        out_ids = {i.id for i in out_of_stock}

        low_stock = [
            i for i in active
            if i.id not in out_ids
            and (
                (i.quantity_mode == QuantityMode.STATUS and i.status == StockStatus.LOW)
                or (
                    i.quantity_mode == QuantityMode.EXACT
                    and i.reorder_threshold is not None
                    and i.quantity is not None
                    and i.quantity <= i.reorder_threshold
                )
            )
        ]

        return HASummaryResponse(
            low_stock_count=len(low_stock),
            out_of_stock_count=len(out_of_stock),
            total_items=len(active),
            low_stock_items=sorted(i.name for i in low_stock),
            out_of_stock_items=sorted(i.name for i in out_of_stock),
        )
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HA Core / local network → /api/ha/summary | Unauthenticated HTTP GET from any client on the local network reaches the endpoint. Per D-03 this is by design (HAOS add-on, not internet-exposed). |
| Browser dashboard → /api/ha/summary | A browser on the local network may fetch the endpoint cross-origin (Lovelace iframe or custom card). CORS headers apply here. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | GET /api/ha/summary (unauthenticated) | accept | Per D-03 the add-on is local-network only; exposed data is household item names and counts — no PII, no credentials. ASVS V4.1 accepts this for local-only admin tooling. |
| T-06-02 | Information Disclosure | CORSMiddleware with allow_origins=["*"] | mitigate | Pair wildcard with `allow_credentials=False` (spec-safe combination) and `allow_methods=["GET"]` (read-only) — prevents cross-origin write and cookie leakage. Enforced in backend/main.py. |
| T-06-03 | Tampering | POST/PUT/DELETE attempts on /api/ha/summary | mitigate | Only `GET` declared on the router; `allow_methods=["GET"]` on CORS. FastAPI returns 405 for other verbs. |
| T-06-04 | Denial of Service | HA sensor poll frequency × table scan | accept | Query is `SELECT * FROM items WHERE archived=0` on a single-household table (<1000 rows). Sub-millisecond at scale. Documented `scan_interval: 60` in Plan 02 README reduces load further. |
| T-06-05 | Spoofing | Clickjacking via iframe on hostile origin | accept | HA-02 requires the app to BE embeddable in Lovelace; blocking iframe embedding would break the requirement. Data is household-only, local-network; clickjacking threat model does not apply (no public internet exposure, no destructive actions without HA ingress auth which bypasses this endpoint entirely). |
</threat_model>
